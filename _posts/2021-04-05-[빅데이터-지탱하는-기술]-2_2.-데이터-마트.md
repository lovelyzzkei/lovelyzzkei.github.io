데이터마트는 데이터레이크에서 데이터를 집계하여 그 데이터를 시각화 도구에게 제공합니다. 주어진 데이터를 크로스 집계하는 데에는 시간이 얼마 걸리지 않기 때문에 빠른 데이터 분석에 있어서 중요한 것은 데이터 집계에 걸리는 속도임을 알 수 있습니다. 이번 포스팅에서는 빠르게 데이터를 집계할 수 있는 방법과 데이터 마트의 기본 구조에 대해 공부해봅시다.

## MPP 데이터베이스

<img src="https://github.com/lovelyzzkei/lovelyzzkei.github.io/blob/master/assets/img/big_data_4.jpg?raw=true">

데이터를 집계하는 구조를 다시 한 번 살펴보면 로우 데이터(raw data)가 있는 `데이터 레이크`와 데이터를 분석하는 `시각화 도구`사이를 중개하는 데이터 마트의 성능이 중요하다는 것을 알 수 있습니다. 데이터 마트가 빠르게 데이터를 필요한 데이터를 데이터레이크 또는 데이터베이스에서 집계를 하여야 사용자가 빠르게 해당 데이터를 가지고 크로스 집계할 수 있기 때문입니다.  

따라서 좋은 데이터 마트는 데이터 집계의 속도가 빨라야합니다. 이를 우리는 **지연이 적어야한다**고 이야기 합니다. 지연율을 적게 하기 위해서 두 가지를 생각해 볼 수 있습니다.   

하나는 데이터레이크(데이터베이스)의 모든 데이터를 메모리 위로 올리는 것입니다. 데이터레이크의 모든 데이터가 메모리 상에 있다면 바로바로 처리가 가능하기 때문에 지연율이 적어질 것입니다. 하지만 데이터 마트에 얼마나 많은 양의 데이터가 집계될지 모르고, 하드웨어상 메모리의 여건이 데이터 양을 따라가지 못할 수도 있습니다. 여기서 하나의 스킬을 사용합니다. 바로 **압축**과 **분산**입니다.   

<img src="https://github.com/lovelyzzkei/lovelyzzkei.github.io/blob/master/assets/img/big_data_5.jpg?raw=true">

데이터베이스 상의 데이터를 가능한 작게 압축하고 그것을 여러 디스크에 분산 처리를 시켜 놓습니다. 그리고 데이터마트가 이를 로드(load)할 때는 멀티코어를 통한 병렬처리로 지연율을 줄입니다. 한마디로 데이터베이스에서 한번 전처리 과정을 거치는 것입니다. 이를 **MPP(Massive Parallel Processing, 대규모 병렬처리) 아키텍처**라고 부릅니다. MPP 아키텍처의 핵심인 압축과 분산에 대해 좀 더 자세히 살펴보죠.

### 압축
데이터 마트에서 데이터를 집계하는 모습을 한번 생각해봅시다. 우리는 어떠한 데이터가 필요하고 어떠한 데이터의 준비를 데이터 마트에 요구할까요? 이전 포스팅의 예시인 판매 데이터를 가지고 한번 생각해봅시다.   
```python
      매출일  점포ID 상품ID 고객ID 금액  상품명 상품 카테고리
0 2017-01-01    11   101  1001  57500  상품A     식료품
1 2017-02-01    12   101  1003  57500  상품A     식료품
2 2017-03-01    12   101  1003  60000  상품A     식료품
3 2017-01-01    11   102  1002   2400  상품B    전자제품
4 2017-02-01    11   102  1002   5800  상품B    전자제품
5 2017-03-01    11   102  1002  12400  상품B    전자제품
```
보기 편하게 두 시트를 합친 결과를 가져왔습니다. 위의 데이터가 데이터베이스에 있다고 가정을 해봅시다. 그리고 우리는 상품A가 팔린 총 금액을 알고 싶다고 가정을 해봅시다. 그렇다면 우리는 위의 표에서 상품명과 금액에 해당하는 **열**이 필요합니다.      
데이터베이스에는 두가지 종류가 있습니다. 바로 행 지향 데이터베이스와 열 지향 데이터베이스입니다. 행 지향 데이터베이스의 경우 데이터를 행으로 처리하기 때문에 위의 예시를 행 지향 데이터베이스로 처리하려고 하면 일단 모든 데이터를 로드해야합니다. 어떤 데이터가 필요한지 행 지향 데이터베이스는 모르기 때문입니다.  
하지만 **열 지향 데이터베이스**의 경우 분석에 필요한 칼럼만을 집계하면 되기 때문에 로드하는 데이터의 양이 줄어들어 지연이 줄어듭니다. 더불어 열 지향 데이터베이스는 동일 칼럼내에 유사한 데이터, 같은 데이터는 압축이 가능하기 때문에 디스크에 부담이 줄어듭니다.   
이러한 장점이 **MPP 아키텍처**와 잘 맞기에 MPP 데이터베이스는 열 지향 데이터베이스를 사용합니다.

### 처리
행 지향 데이터베이스에서는 SQL의 하나의 쿼리가 하나의 스레드에서 실행이 됩니다. 하나의 쿼리를 여러 개의 스레드에 나눠서 작업하는 것이 불가능합니다.   
하지만 MPP 데이터베이스는 열 지향이기 때문에 쿼리를 분해할 수 있습니다. MPP에서는 하나의 쿼리를 다수의 작은 태스크(Task)로 분해하고, 이를 가능한 한 병렬로 실행하여 데이터 집계를 고속화시킵니다.

## 데이터 마트
앞에서 우리는 데이터 마트가 데이터 집계를 빠르게 하기 위하여 MPP 데이터베이스를 사용한다는 것을 공부하였습니다. 이번에는 시각화 도구에서 데이터 분석을 빠르고 정확하게 하기 위한 최적의 데이터 마트의 구조에 대해 알아보겠습니다.

### 테이블의 비정규화
<img src="https://github.com/lovelyzzkei/lovelyzzkei.github.io/blob/master/assets/img/big_data_6.jpg?raw=true">

우리가 BI 도구를 가지고 우리가 원하는 대로 그래프를 그리거나 시각화를 하기 위해서는 데이터 마트에 있는 데이터를 BI 도구에서 우리가 원하는 방향으로 크로스 집계를 해야 합니다. 하지만 데이터베이스에 있는 데이터는 위에서와 같이 정규화가 되어 있습니다. 이 경우 데이터를 우리가 원하는 대로 가공하기가 쉽지 않습니다. 따라서 데이터를 가공하기 위하여 데이터 마트 안에서 테이블을 **비정규화**하는 과정이 필요합니다.   
단어 설명을 조금 하고 이어가겠습니다.  
+ 트랜잭션: 데이터가 생성될 당시에 함께 생성되는 데이터를 일컬어 부른 말, 팩트 테이블이라고도 불림 (불변)
+ 마스터: 트랜잭션에 참고가 되는 데이터, 디멘젼 테이블이라고도 불림 (가변)
+ 스타 스키마: 팩트 테이블을 중심으로 여러 디멘젼 테이블을 결합한 형태

MPP 데이터베이스와 같이 열 지향 스토리지의 등장 이전에는 스타 스키마의 형태로 비정규화를 하였지만 MPP 데이터베이스의 등장 이후 데이터 압축 성능이 높아져 굳이 테이블을 나누어 저장할 필요가 없어졌습니다. 하나의 거대한 팩트 테이블로도 충분해진 것입니다. 이렇게 등장한 것이 모든 테이블을 결합한 **비정규화 테이블**입니다.    
대부분의 경우 데이터 마트는 비정규화 테이블로 만드는 것이 가장 단순하면서 효율적입니다.
